WEB APPLICATION CASE STUDY

CHALLENGE

The existing version of this productivity app was composed of several modules housing individual features. This isolated configuration created a confusing disconnect, making it hard for a user to understand how it was supposed to work. This frustrating and confusing experience proved to be too daunting for the average user to overcome on their own.

Existing design

Busy users expected this productivity tool to be simple and fast to use. In order to deliver this expected experience, I needed to design a dashboard with an instantly understandable workflow of unified features, keeping in mind that our small team needed to recycle as much of the existing design as possible to cut development time. The goal was to pinpoint the exact areas of disconnect and provide both a quick fix and a long term redesign solutions.

PROCESS

My first task was to fully understand where the breakdown was happening for our users. Which design decisions were unnecessary hurdles that we could remove with a new iteration? I went through thousands of customer support inquiries, tagging the issues that were related to the web app.

The overarching issue I discovered related to this project was that the relationship between the two main features (creating “blocks”, then scheduling them) were not understood. The second scheduling step was given greater visual hierarchy than the first step. This created an apparent flow that was directly opposite of the correct sequence needed to successfully use the tool.

I knew that the relationship between those two features needed to be improved. But before I started working at that level, I took a deeper look at each feature individually. For the “blocks feature”, I had discovered in my customer support research that many people did not understand the scope of its function. I needed to more clearly display the standard use cases so that users could fully utilize it. I also found that there were some redundancies in the “scheduling feature” that once removed, would create a simpler experience.

Within this redesign process, I realized that there was an important development opportunity that would be a big win for users, but would not require the development of a new feature. By expanding on more preset values within the block feature, we would be improving on the highest used function within the backend structure we already had in place.

Wireframe and high fidelity mockup for existing block module

After identifying the pain points of each feature individually and their interaction as a whole, I began to sketch out updates that would improve the experience. I learned that our oversimplified design for the settings was actually not making the experience simpler. There were components that needed to be added in order to differentiate tasks and create an automatically intuitive experience.

OUTCOME

By using the same footprint, the changes would not cause a drastic learning curve for existing users. Later, the columns could be reversed in order for the entire dashboard to present the features in complete sequential order.

A unified block and scheduling module with updated device and session modules to more clearly display settings

Ultimately it was decided that having the two features in separate modules was not going to work, so I combined the two features within the same module. This helped convey the relationship between the two features, eliminating all guesswork on how they interact.

With this intuitive flow, users could become less dependent on tool tips and supplementary resources to successfully use the app. The new design would aid in speedier user adoption, creating a larger, more self-sufficient userbase.
